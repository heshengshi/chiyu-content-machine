# 项目执行力框架：从验证到交付的执行系统

> 需求验证通过后，如何把「值得做」变成「做出来了」？这个框架解决的是从认知到行动的最后一公里。

## 为什么需要这个框架

现有框架已经覆盖了从选题到发布的全链路，但有一个关键缺口：

```
选题信号 → 机会挖掘 → 需求验证 → ❓ → 文案生产 → 发布 → 监控
                                  ↑
                            这里断了
```

验证通过的需求，如果没有执行系统，就会变成「知道该做但没做」。这个框架填补这个缺口。

## 一、项目拆解：把「值得做」变成「可执行」

### 1.1 一页纸项目定义

每个验证通过的需求，在动手之前必须写清楚这 7 个问题：

| # | 问题 | 示例 |
|---|------|------|
| 1 | 一句话描述项目 | 用 AI 自动生成小红书图文笔记，月产 100 条 |
| 2 | 目标用户是谁 | 想做小红书但没有内容生产能力的个体创业者 |
| 3 | 用户愿意为什么付费 | 省时间（自动化）+ 省脑力（AI 生成） |
| 4 | 最小可交付版本是什么 | 能自动生成 1 条图文笔记的脚本 |
| 5 | 做到 MVP 需要几天 | 3 天 |
| 6 | 做到 MVP 需要什么资源 | Python + GPT API + 小红书 API |
| 7 | 怎么判断 MVP 成功 | 连续 3 天自动发布，0 人工干预 |

**规则**：7 个问题写不清楚的项目，不开始。写不清楚 = 还没想清楚 = 做了也会返工。

### 1.2 MVP 拆解法

把项目拆成 3 层，从内到外：

```
┌─────────────────────────────────┐
│  V2：完整版（可以以后做）         │
│  ┌───────────────────────────┐  │
│  │  V1：可用版（1-2 周交付）   │  │
│  │  ┌─────────────────────┐  │  │
│  │  │  MVP：最小可交付      │  │  │
│  │  │  （3 天内交付）       │  │  │
│  │  └─────────────────────┘  │  │
│  └───────────────────────────┘  │
└─────────────────────────────────┘
```

**MVP 标准**：能跑通核心流程，哪怕界面很丑、功能很少、只有自己能用。
**V1 标准**：能给 1-3 个真实用户用，能收集反馈。
**V2 标准**：能公开推广，能承载 100+ 用户。

### 1.3 任务原子化

MVP 确定后，拆成原子任务。每个原子任务必须满足：

- **2 小时内能完成**（超过 2 小时 = 拆得不够细）
- **有明确的完成标准**（不是「优化一下」，而是「API 返回 200 且数据格式正确」）
- **可以独立验证**（不需要等其他任务完成才能检查）

示例：

| ❌ 错误拆法 | ✅ 正确拆法 |
|------------|------------|
| 搭建后端 | 1. 初始化 FastAPI 项目 + 健康检查接口 |
| | 2. 接入 GPT API，写 prompt 模板 |
| | 3. 实现生成接口，输入选题输出文案 |
| 做前端 | 1. 创建 HTML 页面 + 输入框 + 生成按钮 |
| | 2. 调用后端接口 + 展示结果 |
| 接入小红书 | 1. 研究小红书发布 API 文档 |
| | 2. 实现自动发布脚本（单条） |
| | 3. 实现批量发布 + 定时 |

## 二、优先级系统：多项目并行的决策框架

### 2.1 ICE 评分法

当有多个验证通过的需求时，用 ICE 评分决定先做哪个：

| 维度 | 含义 | 1 分 | 5 分 | 10 分 |
|------|------|------|------|-------|
| **I**mpact | 做成后的收益 | 小众需求，月入 < 1K | 中等市场，月入 1K-10K | 大众需求，月入 > 10K |
| **C**onfidence | 对判断的信心 | 纯猜测 | 有 2-3 个数据点支撑 | 已有付费用户或强信号 |
| **E**ase | 执行难度 | 需要 2 周+，涉及新技术 | 1 周内，用熟悉的技术 | 3 天内，纯拼装 |

**ICE 分数 = I × C × E**

示例：

| 项目 | I | C | E | ICE | 排序 |
|------|---|---|---|-----|------|
| AI 小红书图文生成器 | 7 | 6 | 8 | 336 | 1 |
| AI 短视频批量剪辑 | 8 | 4 | 3 | 96 | 3 |
| AI SEO 文章代写 | 5 | 8 | 9 | 360 | 2 |

**注意**：SEO 文章代写 ICE 更高但 Impact 更低。如果当前阶段需要现金流，优先选 Impact 高的；如果需要快速验证执行力，优先选 Ease 高的。

### 2.2 精力分配原则

同时进行的项目数量上限 = **2 个**

- **1 个主力项目**（70% 精力）：正在推向 V1/V2 的项目
- **1 个探索项目**（30% 精力）：MVP 阶段的项目

超过 2 个 = 每个都做不好 = 全部失败。

### 2.3 项目状态机

每个项目只有 4 个状态：

```
验证中 → MVP → V1 → V2
  ↓        ↓     ↓
 砍掉    搁置   搁置
```

**状态转换规则**：
- **验证中 → MVP**：ICE 评分 > 200，7 个问题写清楚了
- **MVP → V1**：MVP 跑通 + 有 1 个真实用户反馈
- **V1 → V2**：有 3+ 用户主动使用 + 付费意愿验证
- **→ 砍掉**：MVP 3 天没跑通，或验证数据推翻假设
- **→ 搁置**：不是不行，但现在有更重要的项目。搁置 ≠ 放弃，写清楚搁置原因和恢复条件

## 三、执行节奏：从日到周的操作系统

### 3.1 每日三件事

每天只做 3 件对项目推进有实质贡献的事：

| 时段 | 事项 | 时长 | 标准 |
|------|------|------|------|
| 上午 | 推进主力项目 | 2-3h | 完成至少 1 个原子任务 |
| 下午 | 推进探索项目 / 处理杂事 | 1-2h | 完成至少 1 个原子任务 |
| 晚上 | 输出 + 复盘 | 30min | 写一条验证过的认知或项目进展 |

**反模式**：
- ❌ 一天做 10 件小事，但没有一件推进项目 → 忙碌但无产出
- ❌ 一天只做探索不执行 → 永远在准备，永远不开始
- ❌ 一天只执行不复盘 → 重复犯错，方向偏了不知道

### 3.2 周度检查

每周日做一次项目检查：

| 检查项 | 判断标准 | 行动 |
|--------|---------|------|
| 本周完成了几个原子任务？ | ≥ 5 个 = 正常 | < 5 → 找瓶颈 |
| MVP/V1 进度如何？ | 按计划推进 | 滞后 > 2 天 → 缩减范围 |
| 有没有新的需求信号？ | 扫描信号源 | 高 ICE → 加入探索队列 |
| 有没有项目该砍掉/搁置？ | 看状态机 | 果断砍，不恋战 |
| 本周输出了几条认知？ | ≥ 3 条 | < 3 → 降低输出门槛 |

### 3.3 时间盒（Timeboxing）

每个任务有固定时间盒，时间到了就停：

| 任务类型 | 时间盒 | 超时处理 |
|---------|--------|---------|
| 原子任务 | 2h | 停下来，写清楚卡在哪里，明天继续 |
| 技术调研 | 1h | 1h 还没结论 = 信息不够，先做假设再验证 |
| Bug 修复 | 30min | 30min 修不好 = 写清楚问题，换思路或求助 |
| 文案/内容 | 45min | 先出 60 分版本，迭代比完美重要 |

## 四、执行中的常见陷阱

### 4.1 陷阱清单

| # | 陷阱 | 表现 | 解药 |
|---|------|------|------|
| 1 | **过度设计** | MVP 还没做就在想 V2 的功能 | 只做 MVP 范围内的，其他写进「以后再说」列表 |
| 2 | **技术选型纠结** | 花 2 天选框架，0 天写代码 | 用最熟悉的技术，3 天后如果卡住再换 |
| 3 | **完美主义** | 一个功能改了 5 遍还不满意 | 60 分就发布，用户反馈比自我感觉准 |
| 4 | **范围蔓延** | MVP 做着做着加了 3 个新功能 | 新功能一律进「V2 列表」，MVP 不加 |
| 5 | **孤岛开发** | 做了 2 周没人知道 | 每天输出进展，哪怕只是 1 句话 |
| 6 | **跳过验证** | 觉得「肯定有人要」就直接做 | 回到 need-validation-framework.md，至少做 3 个验证动作 |
| 7 | **沉没成本** | 做了 1 周发现方向不对但舍不得砍 | 砍掉的成本是 1 周，继续的成本是 1 周 × N |

### 4.2 「3 天法则」

**规则**：任何项目，3 天内必须有一个可运行的东西（哪怕很丑）。

- 3 天有东西 → 继续
- 3 天没东西 → 要么拆得不够细，要么技术选型错了，要么这个项目不该现在做
- 3 天没东西且找不到原因 → 砍掉或搁置

这不是偷懒，这是防止「做了 2 周发现方向错了」的灾难。

## 五、项目文档模板

每个项目创建一个文档，用以下模板：

```markdown
# [项目名]

## 一页纸定义
- 一句话：___
- 目标用户：___
- 付费理由：___
- MVP：___
- MVP 天数：___
- 所需资源：___
- 成功标准：___

## ICE 评分
- Impact: _ × Confidence: _ × Ease: _ = _

## 状态
- [ ] 验证中 / [ ] MVP / [ ] V1 / [ ] V2 / [ ] 搁置 / [ ] 砍掉

## 任务列表
### MVP
- [ ] 任务1（预计2h）
- [ ] 任务2（预计1h）
- [ ] ...

### V1（MVP 完成后再填）
- [ ] ...

## 每日进展
| 日期 | 完成任务 | 卡点 | 明天计划 |
|------|---------|------|---------|
| | | | |

## 以后再说（V2+）
- 功能1
- 功能2
```

## 六、与现有框架的衔接

```
topic-signal-framework.md    → 发现选题信号
        ↓
opportunity-framework.md     → 挖掘赚钱机会
        ↓
need-validation-framework.md → 验证需求
        ↓
★ project-execution-framework.md → 拆解项目、排优先级、执行  ★ 本文档
        ↓
script-framework.md          → 生产文案
        ↓
pre-publish-checklist.md     → 发布前检查
        ↓
automation-monitoring-checklist.md → 发布后监控
```

**关键衔接点**：
1. need-validation-framework 的验证结论 → 本框架的「一页纸定义」第 1-3 题
2. 本框架的 MVP 任务 → script-framework 的文案生产
3. 本框架的执行节奏 → community-patterns 的「每日输出」模式
