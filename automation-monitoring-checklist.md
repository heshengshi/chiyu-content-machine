# 自动化发布监控清单

> 发布不是结束，监控才是闭环。自动化发布后需要持续监控，及时发现异常并处理。

## 一、监控指标基线

### 1.1 各平台正常数据基线

建立自己的账号基线，而不是和爆款比。

| 平台 | 指标 | 新号基线（前 30 天） | 稳定期基线 |
|------|------|---------------------|-----------|
| 视频号 | 完播率 | > 15% | > 30% |
| 视频号 | 点赞率 | > 0.3% | > 1% |
| 视频号 | 评论率 | > 0.05% | > 0.2% |
| 小红书 | 收藏率 | > 1% | > 3% |
| 小红书 | 评论率 | > 0.2% | > 0.5% |
| 小红书 | 点击率 | > 3% | > 8% |
| 抖音 | 完播率 | > 20% | > 40% |
| 抖音 | 互动率 | > 1% | > 2% |

**计算公式**：
- 完播率 = 平均观看时长 / 视频总时长
- 点赞率 = 点赞数 / 播放量
- 收藏率 = 收藏数 / 播放量
- 互动率 = (点赞 + 评论 + 收藏 + 分享) / 播放量

### 1.2 异常阈值设定

| 指标 | 正常范围 | 异常阈值 | 可能原因 |
|------|---------|---------|---------|
| 播放量 | > 基线 50% | < 基线 30% | 限流/内容差/发布时间错 |
| 完播率 | > 基线 | < 基线 50% | 内容无聊/开头没钩子 |
| 点赞率 | > 基线 | < 基线 30% | 内容没价值/太硬广 |
| 评论率 | > 基线 | > 基线 300% | 有争议/被挂/意外爆款 |
| 取关率 | < 0.1% | > 0.5% | 内容偏离定位/过度营销 |

## 二、发布后监控时间表

### 2.1 发布后 0-2 小时（黄金监控期）

**监控内容**：
- [ ] 视频是否成功发布（审核通过）
- [ ] 初始播放量是否 > 0
- [ ] 评论区是否正常（无恶意评论）
- [ ] 是否收到平台通知

**异常处理**：
```
审核不通过 → 立即自查 → 修改后重发/放弃
播放量为 0（超过 30 分钟）→ 检查是否限流 → 联系客服/删帖重发
恶意评论 → 及时删除/回复化解
```

### 2.2 发布后 2-24 小时（数据观察期）

**监控内容**：
- [ ] 播放量增长曲线是否正常
- [ ] 完播率是否达标
- [ ] 是否有爆款迹象（播放量突增）
- [ ] 是否有竞品/抄袭出现

**异常处理**：
```
播放量突然停止增长 → 可能限流 → 检查内容/联系客服
完播率低 → 优化开头/调整时长
出现抄袭 → 取证/投诉/联系平台
```

### 2.3 发布后 24-72 小时（长尾观察期）

**监控内容**：
- [ ] 最终播放量是否达到预期
- [ ] 是否有长尾流量（搜索/推荐）
- [ ] 是否带来粉丝增长
- [ ] 评论区的用户反馈

**数据记录**：
```
视频ID | 发布时间 | 72h播放量 | 完播率 | 点赞率 | 评论数 | 备注
-------|---------|----------|--------|--------|--------|-----
xxx    | 06-28 18:00 | 5000 | 35% | 1.2% | 23 | 正常
```

## 三、异常监控脚本逻辑

### 3.1 自动监控脚本结构

```python
# 伪代码示例
def monitor_published_content(content_id, platform):
    # 发布后 30 分钟检查
    check_after(content_id, minutes=30, checks=[
        '审核状态',
        '初始播放量',
        '评论异常'
    ])
    
    # 发布后 2 小时检查
    check_after(content_id, hours=2, checks=[
        '播放量增长曲线',
        '完播率',
        '互动数据'
    ])
    
    # 发布后 24 小时检查
    check_after(content_id, hours=24, checks=[
        '最终数据统计',
        '粉丝增长',
        '竞品监控'
    ])
    
    # 异常触发通知
    if is_abnormal(content_id):
        send_alert(
            channel='飞书/微信',
            message=f'内容 {content_id} 数据异常：{get_abnormal_reason()}'
        )
```

### 3.2 异常判断逻辑

```python
def is_abnormal(content_id):
    data = get_content_data(content_id)
    
    # 播放量异常低
    if data.play_count < baseline.play_count * 0.3:
        return True, '播放量低于基线 70%'
    
    # 完播率异常低
    if data.completion_rate < baseline.completion_rate * 0.5:
        return True, '完播率低于基线 50%'
    
    # 评论率异常高（可能被挂）
    if data.comment_rate > baseline.comment_rate * 3:
        return True, '评论率异常高，可能被挂'
    
    # 取关率异常高
    if data.unfollow_rate > 0.005:
        return True, '取关率异常高'
    
    return False, None
```

## 四、平台限流特征识别

### 4.1 视频号限流特征

| 特征 | 正常 | 限流 |
|------|------|------|
| 播放量增长 | 持续增长 | 突然停止 |
| 推荐来源 | 推荐占比 > 50% | 推荐占比 < 10% |
| 搜索可见性 | 能搜到 | 搜不到 |
| 评论显示 | 正常显示 | 需要审核才显示 |
| 转发效果 | 正常传播 | 转发后他人看不到 |

**检测方法**：
1. 用另一个账号搜索自己的视频
2. 查看播放量增长曲线（是否突然停止）
3. 查看数据详情中的「推荐」占比

### 4.2 小红书限流特征

| 特征 | 正常 | 限流 |
|------|------|------|
| 笔记收录 | 搜索标题能找到 | 搜索标题找不到 |
| 流量来源 | 首页推荐 > 50% | 搜索占比 > 80% |
| 发布后通知 | 有浏览量通知 | 没有浏览量通知 |
| 评论区 | 正常显示 | 只有自己能看到 |

**检测方法**：
1. 搜索自己的笔记标题，看是否出现在结果中
2. 检查是否有「薯条」推广资格（没有 = 可能被限流）
3. 发布后 1 小时看浏览量（< 50 = 异常）

### 4.3 抖音限流特征

| 特征 | 正常 | 限流 |
|------|------|------|
| 播放量 | 正常增长 | 卡在 200-500 |
| 视频审核 | < 30 分钟 | > 2 小时 |
| DOU+ 推广 | 可购买 | 不可购买 |
| 同城显示 | 正常显示 | 不显示 |

**检测方法**：
1. 尝试购买 DOU+（如果提示「不符合条件」= 可能被限流）
2. 发布后看播放量增长（卡在 200-500 = 异常）
3. 检查同城是否显示自己的视频

## 五、异常处理 SOP

### 5.1 发现异常后的标准流程

```
Step 1: 确认异常类型
        ↓
Step 2: 判断严重程度
        - 轻度：数据低于预期但无违规
        - 中度：疑似限流
        - 重度：明确违规/被处罚
        ↓
Step 3: 执行对应措施
        - 轻度：记录数据，调整策略
        - 中度：自查内容，联系客服
        - 重度：删除内容，申诉
        ↓
Step 4: 记录教训
        - 更新「翻车案例库」
        - 更新门禁规则
        ↓
Step 5: 监控后续
        - 观察账号整体权重
        - 观察后续内容表现
```

### 5.2 常见异常处理方案

| 异常类型 | 处理方案 | 预防措施 |
|---------|---------|---------|
| 内容违规 | 删除 → 自查 → 申诉 | 加强敏感词检测 |
| 疑似限流 | 停更 3 天 → 联系客服 → 养号 | 控制发布频率 |
| 数据骤降 | 分析原因 → 优化内容 → 测试 | 建立数据基线 |
| 恶意评论 | 删除/回复 → 拉黑 → 忽略 | 建立黑名单词库 |
| 被抄袭 | 取证 → 投诉 → 公开声明 | 水印 + 原创声明 |

### 6.2 自动生成周报逻辑

```python
def generate_weekly_report(start_date, end_date):
    contents = get_contents_in_range(start_date, end_date)
    
    # 整体数据统计
    total_play = sum(c.play_count for c in contents)
    total_engagement = sum(c.engagement for c in contents)
    new_followers = get_follower_change(start_date, end_date)
    
    # 表现最好/最差
    best = max(contents, key=lambda c: c.play_count)
    worst = min(contents, key=lambda c: c.completion_rate)
    
    # 异常记录
    anomalies = get_anomalies_in_range(start_date, end_date)
    
    # 生成报告
    return format_report({
        'total_play': total_play,
        'total_engagement': total_engagement,
        'new_followers': new_followers,
        'best_content': best,
        'worst_content': worst,
        'anomalies': anomalies
    })
```
