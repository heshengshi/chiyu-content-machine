# Offer 设计框架：从需求到定价到交付的系统方法

> **定位**：你用 `opportunity-framework.md` 找到了需求信号，用 `need-validation-framework.md` 验证了需求真实存在——然后呢？你需要把验证过的需求变成一个**别人愿意付钱的 Offer**。这份框架就是干这个的。
>
> **核心主张**：Offer ≠ 产品。Offer = 痛点 + 承诺 + 交付物 + 价格 + 信任背书。五要素缺一，转化率断崖式下跌。
>
> **与现有框架的关系**：
> - 上游：`opportunity-framework.md`（找信号）→ `need-validation-framework.md`（验需求）
> - 本框架：把验证过的需求变成可卖的 Offer
> - 下游：`first-month-survival-guide.md`（上架找客户）→ `project-execution-framework.md`（标准化放大）

---

## 一、Offer 的五要素模型

```
痛点（Pain）→ 承诺（Promise）→ 交付物（Deliverable）→ 价格（Price）→ 信任（Trust）
     ↓              ↓               ↓                ↓            ↓
  为什么买？     买了能得到什么？   具体拿到什么？    要付多少？    为什么找你买？
```

**每个要素的检验标准**：

| 要素 | 检验问题 | 常见错误 |
|------|---------|---------|
| 痛点 | 用户能不能用一句话说出自己的痛？ | 把"需求"当"痛点"——"我需要一个网站"是需求，"我的客户找不到我所以丢了 3 单"是痛点 |
| 承诺 | 你能不能用一句话说清结果？ | 承诺过程而非结果——"我帮你搭一个自动化流程"是过程，"你每天省 2 小时"是结果 |
| 交付物 | 用户付钱后具体收到什么？ | 交付物模糊——"帮你优化"不是交付物，"3 篇小红书文案 + 7 张配图 + 发布排期表"是 |
| 价格 | 用户觉得值不值？ | 时间定价而非价值定价（见第三节） |
| 信任 | 用户为什么信你？ | 只有"我很专业"，没有证据——案例/样品/免费试用才是信任载体 |

## 二、Offer 设计的六步流程

### Step 1：从需求信号中提取真痛点

**方法**：对验证过的需求，连续问 3 次"为什么"。

```
需求信号："有人想用 AI 做小红书内容"
  → 为什么？因为手动做太慢
    → 为什么慢是问题？因为一天只能发 1 条，流量起不来
      → 为什么流量起不来是问题？因为没有流量就没有客户，没有客户就没有收入

真痛点：有技能/产品但没有流量 → 没有客户 → 没有收入
```

**检验**：真痛点一定和钱、时间、或情绪直接相关。如果问完 3 个"为什么"还停留在"方便""好用"层面，说明痛点不够深，这个需求可能不值得做 Offer。

### Step 2：设计承诺——卖结果，不卖过程

**承诺公式**：

```
[时间范围] + [可量化的结果] + [适用人群]

示例：
❌ "我帮你用 AI 做小红书内容"（卖过程）
✅ "7 天内帮你产出 10 条小红书内容，覆盖你的核心品类"（卖结果）

❌ "我帮你搭自动化工作流"（卖过程）
✅ "3 天内帮你把重复性工作自动化，每天省 2 小时"（卖结果）

❌ "AI 工具推荐"（卖过程）
✅ "帮你筛选出最适合你业务的 5 个 AI 工具，附配置教程"（卖结果）
```

**承诺的边界**：只承诺你能 100% 做到的。宁可承诺保守、交付超预期，也不要承诺激进、交付打折扣。

### Step 3：定义交付物——让用户知道"钱花出去后拿到什么"

**交付物清单模板**：

```markdown
## 交付物清单

### 核心交付物（必须有）
- [ ] 交付物 1：[具体描述]（格式：文件/视频/账号/...）
- [ ] 交付物 2：[具体描述]

### 加值交付物（超预期用）
- [ ] 交付物 3：[具体描述]

### 不包含（明确边界，防止期望膨胀）
- ❌ 不包含 [XX]（如果需要，另行报价）
```

**关键原则**：
1. **交付物必须可数**——"3 篇文案"比"若干文案"好 100 倍
2. **明确不包含什么**比明确包含什么更重要——80% 的客户纠纷来自"我以为包含 XX"
3. **加值交付物是超预期的关键**——承诺 3 个交付 5 个，比承诺 5 个交付 5 个体验好得多

### Step 4：定价——价值定价法

#### 4.1 价值定价 vs 时间定价

```
❌ 时间定价：
  我的时薪 = 200 元
  这个项目需要 10 小时
  报价 = 2000 元
  问题：越熟练 → 时间越少 → 收入越低 → 反向激励

✅ 价值定价：
  这个项目为客户创造的价值 = 20000 元/月（省时间/赚更多/避免损失）
  报价 = 20000 × 10%~30% = 2000~6000 元
  花多少时间是我的效率问题，与客户无关
```

#### 4.2 价值锚定法

用户对价格的感知不是绝对值，而是相对值。你需要给一个"锚"。

```
方法 1：对比替代方案
  "请一个全职运营月薪 8000，我帮你做同样的事只要 2000/月"

方法 2：对比结果价值
  "这套内容方案预计每月带来 5000+ 流量，按 1% 转化、客单价 200 算，
   每月价值 10000+，你只付 2000"

方法 3：对比沉没成本
  "你过去 3 个月手动做内容花了多少时间？按你的时薪算，至少 15000 的机会成本"
```

#### 4.3 定价阶梯

不要只有一个价格。给用户选择，但引导他们选你想要的。

```
基础版：¥XXX    —— 刚好够用，目的是让用户"进门"
标准版：¥XXXX   —— 你真正想卖的这个，性价比最高
高级版：¥XXXXX  —— 价格锚点，让标准版显得划算
```

**定价阶梯的心理机制**：大多数人会选中间档。基础版的存在让标准版显得"不贵"，高级版的存在让标准版显得"划算"。

#### 4.4 新手定价策略（第一月适用）

| 阶段 | 定价策略 | 原因 |
|------|---------|------|
| 第 1 单 | 市场价 30%-50% | 跑通闭环比赚钱重要 |
| 第 2-3 单 | 市场价 50%-70% | 有案例了，可以提价 |
| 第 4-10 单 | 市场价 70%-100% | 有口碑了，接近市场价 |
| 第 10+ 单 | 价值定价 | 有足够案例支撑价值主张 |

**关键**：每次交付后，如果客户满意，下一单必须涨价。不涨价 = 告诉客户你的服务不值更多。

### Step 5：建立信任——没有信任，再好的 Offer 也卖不出去

**信任构建四层**：

```
第 1 层：展示能力（我能做）
  → 免费样品/案例展示/Before-After 对比

第 2 层：展示结果（我做成了）
  → 客户评价/数据截图/第三方背书

第 3 层：降低风险（你不会亏）
  → 不满意退款/先做后付/免费试用

第 4 层：制造稀缺（你不买就没了）
  → 限时/限名额/早鸟价
```

**新手信任策略**（没有案例时）：

1. **做自己的案例**——用自己的业务/账号做实验，拿自己的数据说话
2. **免费帮 1 个人做**——换取真实评价 + Before-After 案例
3. **公开过程**——把做的过程发出来，让潜在客户看到你的专业度
4. **承诺退款**——第一单承诺"不满意全额退款"，把风险从客户转移到自己

### Step 6：写出 Offer 文案

**Offer 文案模板**：

```markdown
## [服务名称]

### 你的痛点
[1-2 句话描述目标用户的痛点，用他们的语言]

### 我的承诺
[1 句话说清结果，包含时间范围 + 可量化结果]

### 你将获得
- ✅ [交付物 1]
- ✅ [交付物 2]
- ✅ [交付物 3]
- 🎁 [加值交付物]

### 价格
- 基础版 ¥XXX：[包含什么]
- 标准版 ¥XXXX：[包含什么] ← 推荐
- 高级版 ¥XXXXX：[包含什么]

### 为什么选我
[1-2 句信任背书：案例/经验/独特方法]

### 保障
[退款承诺/满意度承诺]

### 限量
[名额限制/时间限制]

→ [联系方式/下单链接]
```

## 三、Offer 类型矩阵

不同类型的 Offer 适合不同阶段、不同需求。选错类型 = 事倍功半。

### 3.1 六种 Offer 类型

| 类型 | 定义 | 适合谁 | 定价范围 | 交付周期 | 复制难度 |
|------|------|--------|---------|---------|---------|
| **一次性服务** | 做一次，收一次 | 刚起步，需要快速验证 | 50-500 | 1-3 天 | 低 |
| **套餐服务** | 打包多次交付，一次性收费 | 有重复需求的客户 | 500-3000 | 1-4 周 | 中 |
| **订阅服务** | 按月/季收费，持续交付 | 需要长期支持的用户 | 200-2000/月 | 持续 | 中 |
| **信息产品** | 课程/模板/报告，做一次卖多次 | 有体系化知识可分享 | 9.9-999 | 一次性制作 | 高（前期） |
| **工具产品** | 软件/脚本/自动化，边际成本趋零 | 有技术能力 | 免费-999/月 | 1-4 周开发 | 高（前期） |
| **咨询/诊断** | 面对面/视频分析问题，给方案 | 经验丰富，能快速诊断 | 200-2000/次 | 1-2 小时 | 低 |

### 3.2 选择 Offer 类型的决策树

```
你的需求验证结果是什么？
│
├─ 有人愿意付钱解决一次性问题
│   → 一次性服务（最快跑通闭环）
│
├─ 有人反复遇到同样的问题
│   → 套餐服务（把一次性打包成套餐）
│   → 或 订阅服务（如果需求持续且高频）
│
├─ 你发现了一个很多人有的问题，且解决方案可标准化
│   → 信息产品（做一次卖多次）
│   → 或 工具产品（如果你能写代码）
│
└─ 你对某个领域有深度认知，能快速诊断问题
    → 咨询/诊断（时薪最高，但难规模化）
```

### 3.3 Offer 进化路径

```
一次性服务 → 套餐服务 → 订阅服务
     ↓              ↓
  信息产品 ←───────┘
     ↓
  工具产品（把服务中的重复动作自动化）
```

**关键**：不要跳步。先做一次性服务跑通闭环，再考虑进化。很多人直接想做工具产品，结果做了 2 个月没人用——因为你还没验证过需求。

## 四、Offer 质量检验清单

设计完 Offer 后，用这个清单自检。每一条必须通过，否则回去改。

### 4.1 痛点检验

- [ ] 痛点是否和钱/时间/情绪直接相关？
- [ ] 用户能不能用一句话说出自己的痛？
- [ ] 这个痛点是否足够痛（不解决会怎样？能忍则不痛）？

### 4.2 承诺检验

- [ ] 承诺是否是结果而非过程？
- [ ] 承诺是否可量化（多少/多快/什么效果）？
- [ ] 承诺是否你能 100% 做到？

### 4.3 交付物检验

- [ ] 交付物是否可数（3 篇 vs 若干篇）？
- [ ] 是否明确写了"不包含什么"？
- [ ] 是否有加值交付物（超预期空间）？
- [ ] 交付物是否和承诺直接对应（承诺省 2 小时，交付物是否真能省 2 小时）？

### 4.4 定价检验

- [ ] 定价是否基于价值而非时间？
- [ ] 是否有定价阶梯（至少 2 档）？
- [ ] 基础版价格是否低到目标用户"不用想就能买"？
- [ ] 标准版价格是否在目标用户"觉得值"的范围内？

### 4.5 信任检验

- [ ] 是否有案例/样品/数据展示？
- [ ] 是否有风险逆转（退款/试用）？
- [ ] 是否有稀缺性（限时/限名额）？
- [ ] 如果你是用户，你会买吗？（诚实回答）

### 4.6 整体检验

- [ ] 整个 Offer 能不能用 30 秒说清楚？
- [ ] 你的目标用户看到后会不会觉得"这就是我需要的"？
- [ ] 有没有至少 1 个让用户"哇"的点（超预期/独特方法/稀缺）？

## 五、常见 Offer 设计错误与修正

### 错误 1：卖能力而非卖结果

```
❌ "我会用 AI 帮你做内容"
✅ "7 天产出 10 条小红书爆款内容，帮你从 0 开始获取流量"

修正：把"我能做什么"改成"你能得到什么"
```

### 错误 2：交付物模糊

```
❌ "帮你优化内容策略"
✅ "交付：1 份内容策略文档 + 10 条选题建议 + 3 篇样稿 + 1 份发布排期表"

修正：每个交付物必须有名称、格式、数量
```

### 错误 3：只有一个价格

```
❌ "¥500/次"
✅ "单次 ¥500 / 月度 4 次 ¥1600 / 季度 12 次 ¥4000"

修正：至少提供 2 个选项，引导用户选你想要的那个
```

### 错误 4：没有风险逆转

```
❌ "先付钱，后交付"
✅ "先看样品 → 满意再付 50% → 交付后付尾款 → 7 天内不满意全额退款"

修正：把风险从客户转移到自己。你承担风险 = 你对结果有信心 = 客户更信任你
```

### 错误 5：试图服务所有人

```
❌ "适合所有想做内容的人"
✅ "适合有产品/技能但不会做内容获客的独立创业者"

修正：越窄的定位，越高的转化。服务所有人 = 服务没人
```

### 错误 6：定价太低

```
❌ "9.9 帮你做 XX"（除非是引流品）
✅ 根据价值定价，第一单可以打折，但原价必须是价值定价

修正：低价 = 低价值感知。用户不是买不起，是不确定值不值。解决信任问题，不是降价
```

## 六、Offer 迭代方法

### 6.1 数据驱动的 Offer 迭代

每次 Offer 接触用户后，记录以下数据：

| 指标 | 定义 | 目标值 | 低于目标说明 |
|------|------|--------|------------|
| 曝光→咨询率 | 看到 Offer 的人中，有多少来问 | > 5% | 痛点/承诺不吸引人 |
| 咨询→成交率 | 来问的人中，有多少买了 | > 20% | 信任/价格/交付物有问题 |
| 成交→满意率 | 买了的人中，有多少满意 | > 80% | 交付质量/期望管理有问题 |
| 满意→复购率 | 满意的人中，有多少再买 | > 30% | Offer 类型不对（应该做套餐/订阅） |
| 满意→转介绍率 | 满意的人中，有多少推荐别人 | > 20% | 超预期不够 |

### 6.2 迭代优先级

```
如果曝光→咨询率低 → 先改痛点描述和承诺（Step 1-2）
如果咨询→成交率低 → 先改信任和价格（Step 4-5）
如果成交→满意率低 → 先改交付物和期望管理（Step 3）
如果满意→复购率低 → 先改 Offer 类型（第三节）
如果满意→转介绍率低 → 先加超预期交付物（Step 3 的加值交付物）
```

### 6.3 A/B 测试最小方案

不需要复杂的 A/B 测试工具。用最简方法：

1. **改痛点描述**：同一服务，发 2 条不同文案到不同群，看哪个咨询多
2. **改价格**：同一服务，对不同客户报不同价（前 3 单 vs 后 3 单），看成交率变化
3. **改交付物**：同一服务，一版含加值交付物一版不含，看满意度差异
4. **改信任元素**：一版有案例一版没有，看成交率差异

**每次只改一个变量**，否则不知道哪个因素起了作用。

## 七、从 Offer 到系统化交付

当同一个 Offer 卖了 5 次以上，就该系统化了。

### 7.1 系统化检查清单

- [ ] 交付流程是否写成了 SOP？（每个步骤、每步产出、每步耗时）
- [ ] 是否有交付物模板？（文案模板/报告模板/方案模板）
- [ ] 是否有常见问题 FAQ？（客户常问的 10 个问题 + 标准回答）
- [ ] 是否有质量检查点？（交付前自检的 5 个标准）
- [ ] 是否有自动化空间？（哪些步骤可以用 AI/脚本替代？）

### 7.2 从服务到产品的进化信号

当你发现以下信号时，说明这个服务可以进化成产品：

| 信号 | 含义 | 下一步 |
|------|------|--------|
| 80% 的客户需求相同 | 可以标准化 | 做成信息产品（课程/模板） |
| 交付中有大量重复动作 | 可以自动化 | 做成工具产品（脚本/SaaS） |
| 客户愿意自助完成 | 可以去人工化 | 做成自助式产品 |
| 你不想再亲自交付 | 需要杠杆 | 招人/做培训/做工具 |

## 八、实战案例：3 个 Offer 设计全过程

### 案例 1：AI 小红书内容服务

```
需求信号（来自 opportunity-framework）：
  "有人想用 AI 做小红书内容但不会"

3 层为什么：
  想做内容 → 手动做太慢 → 流量起不来 → 没有客户没有收入

真痛点：有产品/技能但没有内容获客能力 → 没有流量 → 没有收入

承诺：7 天产出 10 条小红书内容，帮你从 0 开始获取精准流量

交付物：
  核心：10 条文案 + 10 组配图 + 1 份发布排期表
  加值：1 份竞品分析报告 + 3 天数据跟踪反馈
  不含：账号运营/投放/客服

定价：
  基础版 ¥299：5 条文案 + 5 组配图
  标准版 ¥599：10 条文案 + 10 组配图 + 排期表 ← 主推
  高级版 ¥999：标准版 + 竞品分析 + 2 周数据跟踪

信任：
  展示：自己账号的数据截图
  风险逆转：不满意退款
  稀缺：每月只接 5 单
```

### 案例 2：AI 工作流搭建服务

```
需求信号（来自 AI Native 群）：
  "有人问怎么用 AI 自动化 XX 流程"

3 层为什么：
  想自动化 → 手动做重复工作 → 浪费时间 → 时间本可以用来做更高价值的事
```
