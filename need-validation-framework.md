# 需求验证框架：从信号到付费的验证管道

> 发现需求信号后，如何快速判断「这是不是真需求」「值不值得做」？

## 为什么需要这个框架

**问题**：
- 从 opportunity-framework 发现了需求信号，但不知道怎么验证
- 想到一个点子就开做，结果做了没人要
- 问了几个朋友都说「好主意」，上线后发现没人付钱

**目标**：
- 用最小成本验证需求真实性
- 区分「嘴上说好」和「愿意付钱」
- 在开发前判断是否值得投入

---

## 一、需求真实性检验清单

### 1.1 三级验证法

| 验证级别 | 方法 | 验证什么 | 成本 | 通过标准 |
|---------|------|---------|------|---------|
| **一级：行为验证** | 观察实际行为 | 有人已经在做这件事了吗？ | 低（搜索即可） | 找到 3+ 个真实案例 |
| **二级：付钱验证** | 找付费证据 | 有人愿意为解决这个问题付钱吗？ | 中（调研） | 找到付费产品/服务 |
| **三级：你也能验证** | 小规模测试 | 你能触达这些人吗？他们愿意为你付钱吗？ | 中高（需执行） | 有人给你付钱/定金 |

### 1.2 一级验证：行为验证

**核心问题**：有人已经在做这件事了吗？

**方法**：
1. 搜索平台搜索关键词，看搜索量
2. 看有没有教程/课程/工具
3. 看社区有没有人在讨论

**判断标准**：
- 搜索量 > 1000/月 → 有需求
- 有付费课程/工具 → 需求被验证过
- 有活跃社区讨论 → 需求存在且有交流需求

**反面信号**：
- 搜索不到相关内容 → 可能是伪需求，或者你关键词错了
- 只有提问没有回答 → 可能是真需求但没人解决（机会）
- 只有免费内容没有付费 → 可能不愿意付钱

### 1.3 二级验证：付钱验证

**核心问题**：有人愿意为解决这个问题付钱吗？

**方法**：

| 方法 | 操作 | 找到什么 |
|------|------|---------|
| **找付费产品** | 搜索「[问题] 工具」「[问题] 课程」 | 已有竞品 |
| **找付费服务** | 搜索「[问题] 代做」「[问题] 咨询」 | 服务提供者 |
| **看定价** | 看竞品/服务的价格区间 | 市场接受价 |
| **看评价** | 看付费用户的评价 | 用户真正在乎什么 |

**判断标准**：
- 有付费产品且活得不错 → 市场已验证，可以做差异化
- 有付费服务但没产品 → 产品化机会
- 没有任何付费 → 需求可能不够痛，或用户没钱

### 1.4 三级验证：你也能验证

**核心问题**：你能触达目标用户吗？他们愿意为你付钱吗？

**方法**：

| 方法 | 执行方式 | 成本 | 有效信号 |
|------|---------|------|---------|
| **预售测试** | 做落地页，挂价格，看有没有人下单 | 低 | 有人下单/留联系方式 |
| **定金测试** | 说「想做这个，感兴趣的先付定金」 | 低 | 有人付定金 |
| **手动服务** | 先手动帮人解决问题，看有没有回头客 | 中 | 有复购/转介绍 |
| **等待名单** | 做个等待名单，看有多少人注册 | 低 | 注册人数 > 预期 |

**关键原则**：
- 不要问「你会买吗」——嘴上说会买的人 90% 不会付钱
- 要让人付钱行为——定金、预付款、等待名单
- 如果没人付钱，说明需求不够痛或信任不够

## 二、验证失败的常见模式

### 2.1 伪需求识别

| 伪需求类型 | 特征 | 真实原因 |
|-----------|------|---------|
| **「如果有就好了」** | 用户说想要，但不愿付钱 | 不够痛，只是锦上添花 |
| **「能不能便宜点」** | 问功能很多，问到价格就犹豫 | 问题不够痛或用户没钱 |
| **「帮个忙呗」** | 找免费方案，不愿意付费 | 没意识到价值或习惯白嫖 |
| **「试试看吧」** | 没有紧迫感，可以无限期等 | 不是刚需，只是好奇 |
| **「功能能不能再加」** | 永远不满意，想要更多 | 想要的是更便宜更多，不是你的方案 |

### 2.2 验证成功的判断标准

**真需求信号**：
- 有人主动问你「多少钱」「怎么买」
- 有人愿意付定金/预付款
- 有人已经在用替代方案（手动、Excel、付费竞品）解决
- 有人抱怨现有方案贵/难用/不满足
- 有人催你「什么时候能做好」

**嘴上说好但不是真需求**：
- 「挺好的」「加油」「期待上线」——没有承诺
- 「如果有免费版我就用」——不愿意付钱
- 「功能能再加XX吗」——想要更多但不愿多付钱
- 「我帮你问问朋友」——推脱，不是真的需要

## 三、需求验证执行清单

### 3.1 验证前准备

```
□ 清晰定义问题：用户遇到什么问题？现在怎么解决的？
□ 定义目标用户：谁会遇到这个问题？他们在哪？
□ 定义成功标准：多少人付钱算验证成功？
□ 定义验证时限：多久没结果就算失败？
```

### 3.2 验证执行

```
【Day 1-2】一级验证：行为验证
□ 搜索关键词，记录搜索量
□ 找教程/课程/工具，记录竞品
□ 看社区讨论，记录讨论热度

【Day 3-4】二级验证：付钱验证
□ 找付费产品/服务，记录价格区间
□ 看付费用户评价，记录真正痛点
□ 判断市场是否已验证

【Day 5-7】三级验证：你也能验证
□ 制作验证材料（落地页/等待名单/预售文案）
□ 触达目标用户（发到社区/群/私聊）
□ 记录行为数据（多少人询问/下单/付定金）

【Day 8】判断
□ 达到成功标准 → 值得做，进入设计阶段
□ 未达标但有信号 → 调整方案，延长验证
□ 完全没信号 → 放弃或换方向
```

### 3.3 验证结果模板

```markdown
## 需求验证报告：[需求名称]

### 一级验证：行为验证

- 社区讨论：[活跃/一般/很少]

### 二级验证：付钱验证

- 结论：[市场已验证/有产品化机会/付费意愿不明]
```

## 四、验证过程中的常见错误

### 4.1 问错问题

| 错误问法 | 正确问法 | 为什么 |
|---------|---------|--------|
| 「你觉得这个功能有用吗？」 | 「你现在怎么解决这个问题的？」 | 问有用会得到「有用」，问解决方式得到真实行为 |
| 「你会买吗？」 | 「多少钱你能接受？」 | 问买不买会得到「会」，问价格得到真实预算 |
| 「这个产品怎么样？」 | 「你最不满意现有方案的什么？」 | 问怎么样得到客套话，问不满得到真痛点 |
| 「需要这个功能吗？」 | 「如果这个功能明天下线，你会怎么办？」 | 问需要得到「需要」，问下线得到真实依赖度 |

### 4.2 找错人

**错误**：问朋友、问同行、问不是目标用户的人

**正确**：
- 问已经在用类似产品的人
- 问已经在付费解决问题的人
- 问主动搜索解决方案的人

**检验方法**：你问的人，上一次为这个问题付钱是什么时候？
- 如果是「从没付过」→ 找错人了
- 如果是「很久以前」→ 可能不是目标用户
- 如果是「最近/正在付费」→ 找对人了

### 4.3 信错信号

| 信号 | 表面判断 | 真实判断 |
|------|---------|---------|
| 很多人点赞 | 需求很火 | 可能只是觉得有趣，不代表愿意付钱 |
| 很多人说「想要」 | 需求验证了 | 要看多少人愿意付钱 |
| 有人问「多少钱」 | 有付费意愿 | 只是询价，要等真正付钱才算 |
| 竞品很多 | 市场很大 | 可能已经是红海，也可能是验证过的好市场 |
| 没有竞品 | 蓝海机会 | 更可能是伪需求或无法变现 |

## 五、从验证到决策

### 5.1 验证成功后的决策

```
验证成功（有人付钱）
    ↓
判断市场规模
    ├─ 大市场 → 做标准化产品
    ├─ 小市场 → 做高客单价服务
    └─ 不确定 → 先做 MVP，边做边测
```

### 5.2 验证失败后的决策

```
验证失败（没人付钱）
    ↓
判断失败原因
    ├─ 问题不够痛 → 换问题/换人群
    ├─ 找错人群 → 换触达渠道
    ├─ 方案不对 → 换方案
    └─ 价格不对 → 调整价格/价值感
```

### 5.3 验证中断后的决策

```
验证中断（没时间/没资源/找不到用户）
    ↓
判断是否值得继续
    ├─ 市场已验证（有竞品活得不错）→ 继续
    ├─ 市场未验证 → 先验证市场，再继续
    └─ 市场验证失败 → 放弃
```

### 验证三问
1. 有人已经在做这件事了吗？（行为验证）

- 搜索不到相关内容
- 没有任何付费产品/服务
- 没人愿意付定金/预付款
- 只有人说「好」「期待」但没人付钱

- 有竞品活得不错
- 有用户抱怨现有方案
- 有人主动问「多少钱」「怎么买」
